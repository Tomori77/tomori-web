# AGENTS.md

## 目的
本文件为 AI 编码助手（如 ChatGPT、Copilot 等）提供项目修改准则，确保在添加或修改功能时**不影响其他已有功能**，并能**进行自我验证**。

## 项目技术栈概要
- **运行时**：Cloudflare Workers
- **框架**：Hono (路由、中间件)
- **数据库**：Cloudflare D1 (SQLite)
- **对象存储**：Cloudflare R2
- **认证**：JWT (`jose` 库)，密码哈希 `bcryptjs`
- **前端**：原生 HTML/CSS/JS (SPA 模式)，无构建工具，轻量路由

## 核心原则

### 1. 编码前思考
- 明确假设，不确定时询问而非猜测。
- 存在歧义时，列出多种解释，不默默选定单一方案。
- 如果任务有明显更简单的做法，直接指出优化思路。
- 发现代码矛盾、逻辑不一致时及时暂停，请求信息澄清。

### 2. 简洁优先
- 用最少的代码解决问题，拒绝冗余实现。
- 不为一次性需求创建抽象层、复杂架构。
- 不盲目增加扩展性、可配置性，应对“未来可能用到”的场景。
- 若代码可大幅精简，主动重写优化。
- 校验标准：以资深工程师视角判断，代码若过于复杂，立即简化。

### 3. 精准修改
- 仅修改与当前任务直接相关的代码内容。
- 不顺手优化相邻代码、注释、排版格式。
- 不重构原本可以正常运行的代码模块。
- 严格匹配项目现有代码风格，保留原有编码习惯。
- 因本次修改产生的无效导入、废弃变量，可直接删除。
- 发现项目中原有的死代码、冗余内容，仅做文字提醒，不擅自删除。

### 4. 目标驱动执行
- 执行任务前，定义清晰、可落地的成功标准。
- 将“修复Bug”转化为：编写用例复现问题，再调试至用例正常通过。
- 将“新增校验功能”转化为：针对异常输入编写测试用例，保证全部通过。
- 将“代码重构”转化为：完成重构后，确保原有所有测试用例正常运行。
- 多步骤复杂任务，先输出简短执行计划，同时标注每一步的验证方式。

### 5. 隔离性
- 每次修改应局限在最小范围，只修改与需求直接相关的文件。
- 不随意重构无关代码，不改变全局样式/工具函数签名，除非有明确理由且已评估影响。

### 6. 可验证性
- 所有变更必须能在本地验证通过（`wrangler dev`）。
- 对于 API 变更，需通过浏览器或 curl 请求验证返回状态码和响应体。
- 对于前端变更，需在浏览器中实际点击和操作确认交互正常。

### 7. 向后兼容
- 数据库表结构变更必须通过**新增迁移文件**进行，不修改已有迁移。
- API 响应格式不可随意删除字段，如需修改，应做加法或版本控制。
- 前端路由和导航菜单修改要保持原路径可访问（或设置重定向）。

## 开发工作流

当你被要求添加新功能或修改现有功能时，请严格遵循以下步骤：

### 步骤 1：理解现有结构
- 阅读相关路由文件 (`worker/router/`)、服务文件 (`worker/services/`)、前端页面 (`public/js/pages/`)。
- 查阅 `schema.sql` 或最新迁移文件，确认数据表结构。
- 确认权限模型 (`requireRole` 中间件) 和当前用户角色划分。

### 步骤 2：影响分析
在动手前，列出可能受影响的模块：
- **API 端点**：新增/修改路由是否与现有路由冲突？是否影响认证中间件？
- **数据库**：是否需要新建表或修改表？若需 ALTER，务必提供迁移 SQL。
- **前端页面**：修改组件是否在其他页面复用？导航栏是否需要更新？
- **权限**：新功能需要什么角色？是否会意外暴露给低权限用户？
- **存储**：R2 路径变更是否影响已有文件引用？D1存储库是否有增删改？

### 步骤 3：实现变更
- 遵循现有代码风格（见下文代码风格部分）。
- 新增文件要放在正确目录（路由放 `worker/router/`，服务放 `worker/services/`，前端页面放 `public/js/pages/` 等）。
- 所有 API 处理必须包含错误捕获，返回统一的 JSON 错误格式 `{ "error": "描述" }` 和合适的 HTTP 状态码。
- 涉及权限的端点必须使用 `authMiddleware` 和 `requireRole()`。
- 如果修改了导航菜单配置，必须同步考虑 `settings` 表中 `nav_items` 的默认值及更新逻辑。

### 步骤 4：验证变更
- **本地运行**：在项目根目录执行 `wrangler dev`，确保 Worker 启动无报错。
- **API 测试**：
  - 使用 `curl` 或浏览器开发者工具发送请求，验证新端点或修改后的端点行为。
  - 测试正常情况、缺少认证、权限不足、参数错误等情况。
- **前端测试**：
  - 打开浏览器访问 `http://localhost:8787`（或指定端口），浏览受影响页面。
  - 测试所有交互：点击按钮、提交表单、路由跳转、导航菜单正确渲染。
  - 检查控制台无 JS 错误，网络请求状态正常。
- **回归验证**：选取 2-3 个核心功能（如文章列表加载、登录流程、管理后台审核）快速过一遍，确保未被破坏。

## 验证清单

在提交变更前，确认以下项目：

- [ ] 无 TypeScript 编译错误，`wrangler dev` 正常启动。
- [ ] 新 API 端点返回正确的 Content-Type（`application/json` 或 `text/html` 等）。
- [ ] 认证和授权正确：无 token 返回 401，低角色访问高权限接口返回 403。
- [ ] 数据库操作未使用字符串拼接 SQL，全部使用预编译语句（`?` 占位符）。
- [ ] 文件上传大小限制生效（超过 2MB 返回 413）。
- [ ] 文章内容大小限制生效（超过 256KB 拒绝）。
- [ ] 前端新页面可被路由匹配，不出现 404 或空白。
- [ ] 管理后台相关功能仅管理员可见且可用。
- [ ] 导航栏动态渲染正常，新增页面可通过修改 `nav_items` 展示。
- [ ] 日志记录正确写入 `audit_logs`（审核、权限变更等敏感操作）。
- [ ] 工具页的 HTML 运行在 `iframe sandbox` 内，不污染主页面。

## 代码风格与约定

- **Worker 端**：使用 TypeScript，定义请求/响应类型，保持函数简短。
- **前端**：使用 ES 模块 (`<script type="module">`)，类名小写加连字符（如 `glass-card`），CSS 变量定义颜色和效果。
- **命名**：数据库 snake_case，JavaScript 驼峰命名。
- **导入顺序**：第三方库 → 内部模块 → 类型定义。
- **注释**：复杂逻辑添加必要注释，解释“为什么”而非“做什么”。

## 关键注意事项

1. **数据库迁移**：任何表结构更改必须创建新的迁移文件（如 `migrations/002_add_tools_table.sql`），并更新 `schema.sql` 作为最新参考。
2. **权限**：绝不能将管理端点暴露给未经授权的角色。所有 `/api/admin/*` 路由必须通过 `requireRole(3)` 或 `requireRole(4)`。
3. **JWT 密钥**：从环境变量 `JWT_SECRET` 读取，绝不允许硬编码在代码中。
4. **R2 路径**：上传文件生成的 key 必须遵循 `类别/关联ID/日期/文件名` 的人类可读规范。
5. **前端路由**：使用 hash 模式（`/#/path`）或 history 模式时要确保 Worker 对未知路径返回 `index.html`。
6. **错误处理**：API 路由必须用 `try-catch` 包裹，避免 Worker 崩溃，并返回有意义的错误信息。

## 总结
遵循以上指南，你将能够安全、独立地为项目添加功能，同时保持原有系统的稳定。任何不确定的地方，应优先保持现有行为不变，并说明需要人工确认的部分。
